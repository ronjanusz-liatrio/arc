## Editorial Decisions

- **Validation glob pattern.** The spec text uses `{dir}/*-validation-*.md` (LG-6 step 2). I matched that pattern verbatim, even though existing LG-3/LG-4/LG-5 use `{dir}/NN-validation-*.md`. The spec's choice of the broader glob is intentional - the spec body in 18-spec-arc-status-spec-completion.md FR text reads "glob `{dir}/*-validation-*.md`" - so I preserved it without normalizing toward the LG-3/4/5 form. This matches the working-tree convention where validation files for spec 17 (the canonical orphan example) match `*-validation-*.md` (e.g., `17-validation-1-...md`, `17-validation-2-...md`) and the broader glob is more inclusive without false positives.
- **Skipping language.** I used "skip - this spec is already linked to a backlog entry and is handled by LG-3, LG-4, or LG-5 as appropriate" for predicate 5 to make the intent unambiguous: a linked spec is *not* an orphan, and its lifecycle gaps are owned by the spec-scoped detections that already exist. This phrasing avoids the appearance that LG-6 silently overlaps with LG-5 when a PASS-validated spec is linked but not yet shipped.
- **Trailing-slash normalization wording.** I matched the spec's exact phrase "after whitespace trim and trailing-slash normalization" for predicate 4. This phrase is reused in the WL-3 spec-to-idea linkage rule (Step 6.6 item 3), keeping LG-6 detection language consistent with the existing spec-to-idea linkage convention.
- **No reference to the postamble.** LG-6 detection is intentionally silent about scope tagging - tagging is Step 6.6's job (T01.2) and adding any forward reference here would either pre-empt T01.2 or create a stale cross-reference if T01.2's edit shape changes.
