## Editorial Decisions

- **Step 14 positioning.** The task description offered two options: (a) insert as Step 14 after Step 13, or (b) add a separate phase that runs unconditionally on every `/arc-sync` invocation regardless of mode. I chose option (a) — insert as Step 14 — because it preserves the linear `### Step N:` numbering convention used throughout the rest of the file and makes the invocation graph (Step 6 -> Step 14, Step 13d -> Step 14) readable without introducing a separate "phase" abstraction the file does not otherwise use. Step 14's body explicitly states the unconditional-run intent ("This step runs unconditionally on every `/arc-sync` invocation, after the README sync writes complete...") so the option-(b) semantic is preserved within the option-(a) structure. The injection-mode early-exit branch in Step 2a (user cancels) is the one explicit non-run case, called out at line 882.
- **Migrated vs refreshed label rule.** The task description allowed implementer's choice for distinguishing the two; I picked legacy field-name detection — checking the prior block content for `**Backlog:**`, `**Current Wave:**`, `**Phase:**`, `**Primary Personas:**`, or `**Vision:**`. These are the exact field labels removed from the schema in T01.1, so any block containing them is conclusively a legacy live-block. Anything else is treated as `refreshed`. The bootstrap-protocol's idempotency contract is preserved either way — the bytes written are identical.
- **Bootstrap-protocol referencing.** Per the task description ("do not duplicate the template inline here"), Step 14 explicitly forbids inlining the literal template bytes. The Step 14 body references `bootstrap-protocol.md` three times (canonical content, insertion algorithm, References list). Proof 5 verifies via grep that none of the static-template phrases appear in SKILL.md.
- **Step 14e (Never-rules) placement.** I added a sub-section 14e enumerating four Never-rules specific to Step 14 (do not create CLAUDE.md, do not modify TEMPER:/MM:, do not move marker positions, do not duplicate the template inline). These mirror the existing Critical Constraints style without polluting that top-level list with details only relevant to one step.
- **Decision table format (14a).** The decision logic is presented as a three-row Markdown table (State / Detection Rule / Action) for at-a-glance scanability rather than as a prose-only paragraph. This mirrors other tables already in SKILL.md (e.g., Step 2 mode-detection table at line 173).
