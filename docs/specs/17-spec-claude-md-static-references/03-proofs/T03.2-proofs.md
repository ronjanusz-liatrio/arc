## Editorial Decisions

- **Insertion position.** The pre-existing `produces.files` ordering was not strictly alphabetical — it grouped root-level files first (`README.md`) then `docs/`-prefixed files (alphabetical within group: `docs/BACKLOG.md`, `docs/ROADMAP.md`) then the deepest path last (`docs/skill/arc/waves/{wave-slug}.md`). I preserved that depth-grouped convention by placing `CLAUDE.md` in the root-files group. Within the root-files group I placed it before `README.md` (alphabetical: `C` before `R`). This satisfies the task description's directive: "keep alphabetical/logical ordering consistent with the rest of the file."
- **No frontmatter field other than `produces.files` was touched.** `name`, `description`, `user-invocable`, `allowed-tools`, `requires`, `produces.artifacts`, `produces.state-transition`, `consumes`, and `triggers` are byte-identical to their pre-edit state. Verified by `git diff --stat` showing 1 insertion and 0 deletions, and by reading the diff hunk which is bounded entirely within `produces.files`.
- **No corresponding `produces.artifacts` entry was added.** The frontmatter schema treats `produces.artifacts` as a list of *artifact identifiers* (e.g., `README`, `BACKLOG`, `ROADMAP`, `wave-archive`) — these are conceptual artifacts that exist independently of the file paths in `produces.files`. The `ARC:product-context` block is a managed *section within* `CLAUDE.md`, not a freestanding artifact, and it has no analogue in `requires.artifacts` or `consumes.from[].artifact` elsewhere in the repo. Adding an artifact entry would imply downstream skills could `consumes.from { skill: /arc-sync, artifact: CLAUDE-product-context }`, which is not a real consumer relationship in the system. The task description specifically directed only the `produces.files` list, so the scope is correct.
