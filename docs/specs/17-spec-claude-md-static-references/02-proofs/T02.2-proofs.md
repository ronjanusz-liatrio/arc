## Editorial Decisions

- The task description (item 10) requires "Run `/arc-sync` if the project's CLAUDE.md product-context block needs creation or refresh." inside Step 7 (renumbered Confirm). The spec proof grep, however, forbids the literal substring "product-context" inside numbered execution steps. To honor both: the Step 7 closing line uses the spaced phrase "product context block" (dropping the hyphen) while preserving the user-facing intent verbatim. This mirrors T02.1's resolution for the equivalent /arc-wave Step 10 handler text. The hyphenated form remains in the Overview (line 40), which is outside the numbered execution steps and therefore allowed.
- The Overview's redirect sentence reads "This skill does not write to `CLAUDE.md`; run `/arc-sync` if the project's `CLAUDE.md` product-context block needs creation or refresh." This explicitly negates any write claim while still naming the marker block by its canonical hyphenated identifier, which a reader navigating from CLAUDE.md (where the marker is `<!--# BEGIN ARC:product-context -->`) can recognize.
- The frontmatter description (line 3) was left unchanged. Pre-task it did not mention CLAUDE.md or product-context, so no edit was required to honor the "remove writes from frontmatter description" intent — the post-task description still accurately summarizes the skill (transitions ideas, archives to wave file, removes from BACKLOG, cw-validate gate).
- The References list dropped only the `skills/arc-wave/references/bootstrap-protocol.md` bullet. The other four bullets (ship-criteria, wave-archive, idea-lifecycle, cross-plugin-contract) remain — they are unrelated to product-context refresh.
- The Critical Constraint about "NEVER modify cw-validate proof artifacts — read-only verification only" was retained — it concerns cw-validate proofs, not CLAUDE.md, and applies independent of this task.
