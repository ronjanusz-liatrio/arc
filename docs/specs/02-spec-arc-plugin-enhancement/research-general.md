> Generated by cw-research on 2026-04-07

## Summary

- **Arc is a markdown-only Claude Code plugin (v0.2.0)** providing three skills (`/arc-capture`, `/arc-shape`, `/arc-wave`) that manage a product idea lifecycle from raw capture through spec-ready handoff to the SDD pipeline.
- **File-based state machine**: All state lives in `docs/BACKLOG.md` with explicit status transitions (`captured` -> `shaped` -> `spec-ready` -> `shipped`) and two allowed backward transitions. No external APIs, databases, or network calls.
- **Three-plugin ecosystem**: Arc (product direction) feeds into `claude-workflow` (SDD pipeline) and reads phase constraints from `temper` (engineering maturity). Plugins communicate exclusively through markdown files and managed CLAUDE.md sections.
- **Phase-aware templates**: Four templates (VISION, CUSTOMER, ROADMAP, BACKLOG) scale content requirements across 7 Temper maturity phases (Spike -> Maturity), with Liatrio brand colors in Mermaid diagrams.
- **Quality baseline**: 48 Gherkin scenarios across 7 feature files, 2 completed proof cycles, all 7 demoable units implemented. No CI/CD automation; error-path scenario coverage is weak.

## Architecture & Patterns

### Design Patterns

1. **File-Based State Machine**: Ideas in `docs/BACKLOG.md` progress through 4 explicit lifecycle stages with field accumulation at each stage. Two backward transitions allowed (Shape->Capture, Spec-Ready->Shape).

2. **Parallel Subagent Analysis**: `/arc-shape` launches 4 concurrent Agent subagents — Problem Clarity, Customer Fit, Scope Boundaries, Feasibility — each reading different context files and returning structured ratings + gaps.

3. **Phase-Aware Templates**: All 4 templates define required sections per Temper maturity phase (Spike through Maturity). Foundation is the "full capability" gate; MVP+ phases are "maintained" states.

4. **Managed Markdown Sections**: HTML comment markers (`<!--# BEGIN ARC:product-context -->`) in CLAUDE.md enable multi-plugin coexistence with TEMPER: and MM: namespaces.

5. **Brief-as-Contract**: The 7-field brief format (`references/brief-format.md`) is Arc's output and claude-workflow's input. A 19-point validation checklist gates promotion.

### Module Boundaries

| Module | Responsibility | Communication |
|--------|---------------|---------------|
| `arc-capture` | Fast idea entry | Writes to `docs/BACKLOG.md` |
| `arc-shape` | Interactive refinement | Reads BACKLOG + context files, updates BACKLOG, creates `shape-report.md` |
| `arc-wave` | Delivery planning | Reads all artifacts, updates BACKLOG + ROADMAP, creates `wave-report.md`, injects CLAUDE.md |
| `templates/` | Phase-graduated scaffolding | Read by skills when creating artifacts |
| `references/` | Authoritative definitions | Read by skills for lifecycle rules, brief format, wave sizing |

### Key Abstractions

- **Idea Lifecycle** (`references/idea-lifecycle.md`): 4-stage state machine (Capture -> Shape -> Spec-Ready -> Shipped) with entry/exit criteria and data field progression per stage.
- **Brief Format** (`references/brief-format.md`): 7-section structure (Problem, Proposed Solution, Success Criteria, Constraints, Assumptions, Wave Assignment, Open Questions).
- **Wave Planning** (`references/wave-planning.md`): Precedence rules (dependencies first, priority ordering, risk-first, theme coherence) and phase-based sizing.

### Conventions

- **Context markers**: Each skill begins responses with `**ARC-CAPTURE**`, `**ARC-SHAPE**`, or `**ARC-WAVE**`
- **SKILL.md structure**: YAML frontmatter (name, description, user-invocable, allowed-tools) -> Overview -> Critical Constraints -> Numbered Process Steps -> References
- **Timestamps**: ISO 8601 throughout (`2026-04-07T14:30:00Z`)
- **Anchors**: Lowercase, hyphens for spaces, strip special characters
- **Commits**: Conventional Commits (`type(scope): description`)

### Deep-Dive Findings

- **Subagent construction**: Each dimension agent receives the captured stub + relevant context files (CUSTOMER.md, VISION.md, ROADMAP.md, management-report.md) and returns a structured assessment with ratings (High/Medium/Low, Strong/Moderate/Weak), gap lists, and suggested field content. Aggregation merges gaps, identifies critical ones, and pre-populates brief fields.
- **Bootstrap protocol**: CLAUDE.md insertion uses a 3-priority algorithm (after last TEMPER: section -> before Snyk block -> EOF). Conflict resolution skips positions that would nest ARC markers inside other namespaces.
- **Error handling**: Arc-capture creates BACKLOG if absent. Arc-shape fails gracefully if no captured ideas exist. Arc-wave warns (doesn't create) if CLAUDE.md is absent, creates ROADMAP/VISION/CUSTOMER from templates if absent.
- **Wave report**: Contains wave goal, selected ideas table, Temper context, per-idea `/cw-spec` handoff instructions, and backlog status snapshot.

## Dependencies & Integrations

### External Dependencies

| Dependency | Type | Purpose |
|-----------|------|---------|
| `temper` plugin | Companion | Engineering maturity phases, `docs/management-report.md` for wave sizing constraints |
| `claude-workflow` plugin | Downstream | SDD pipeline (`/cw-spec` -> `/cw-plan` -> `/cw-dispatch`) receives spec-ready briefs |
| Claude Code | Runtime | Plugin host, SKILL.md execution, tool framework |

No external APIs, databases, webhooks, or network calls. All operations are local file I/O.

### Integration Points

**Temper integration (6 touchpoints):**
1. Reads `docs/management-report.md` for Temper phase + hard gate failures (arc-shape feasibility, arc-wave scope)
2. Phase-based wave sizing constraints (`references/wave-planning.md`: Spike=1, Foundation+=3-5)
3. Hard gate failure handling (reduce scope, add stabilization work)
4. CLAUDE.md managed section coexistence (ARC: namespace respects TEMPER: boundaries)
5. Phase-graduated templates align with 7 Temper maturity phases
6. `/temper-audit` feedback loop after SDD pipeline completion

**Claude-workflow handoff (3 mechanisms):**
1. Brief format (`references/brief-format.md`) is the contract between Arc and `/cw-spec`
2. Wave report (`docs/wave-report.md`) includes per-idea `/cw-spec` starter prompts
3. Pipeline sequence: `/arc-wave` -> `/cw-spec` -> `/cw-plan` -> `/cw-dispatch`

**Liatrio brand integration**: Mermaid diagrams in all references, templates, and README use consistent brand colors (#11B5A4 teal, #E8662F orange, #1B2A3D dark blue) with Inter font.

### Internal Data Flow

```
/arc-capture
  writes -> docs/BACKLOG.md (status: captured)

/arc-shape
  reads -> docs/BACKLOG.md, CUSTOMER.md, VISION.md, management-report.md

/arc-wave
  reads -> docs/BACKLOG.md, ROADMAP.md, VISION.md, CUSTOMER.md, management-report.md
  writes -> docs/ROADMAP.md, docs/BACKLOG.md (status: spec-ready),
            docs/wave-report.md, docs/VISION.md*, docs/CUSTOMER.md*, CLAUDE.md
  (* = created from template if absent)
```

### Request/Response Shapes

7 required sections: Problem, Proposed Solution, Success Criteria, Constraints, Assumptions, Wave Assignment, Open Questions. Validated by 19-point checklist (`skills/arc-shape/references/brief-validation.md`).

Each dimension returns: rating level, specific findings, gap list, suggested field content. Ratings: Problem Clarity (High/Medium/Low), Customer Fit (Strong/Moderate/Weak/No fit), Scope (Small/Medium/Large/Too large), Feasibility (Ready/Caveats/Needs spike/Not feasible).
