## What Changed

Step 6 now has a postamble sub-step (Step 6.6) that documents, in the skill's operational procedure, how each of the five detected gap types plus skipped checks get a `scope` field assigned before the Lifecycle Gaps table is emitted:

1. LG-1 (Captured → Shaped) — subject = backlog idea title; look up in wave-linked set.
2. LG-2 (Shaped → Spec) — subject = backlog idea title; look up in wave-linked set.
3. LG-3 (Spec → Plan) — subject = spec directory; resolve via BACKLOG `Spec:` field to a backlog idea, then look up.
4. LG-4 (Plan → Validation) — same spec-to-idea resolution as LG-3.
5. LG-5 (Validation → Shipped) — subject = the backlog idea whose `Spec:` field matched; look up.
6. Skipped checks — `scope = --` placeholder; ignored by renderer and precedence.
7. No active wave — all gaps tagged `backlog-only`; scope field functionally unused.

The postamble also delegates authoritative algorithm details to `status-dimensions.md` WL-3 (Gap Scope Tagging).

The rendered output format of the Lifecycle Gaps table is unchanged by this sub-task — the conditional Scope column rendering is the responsibility of T01.4 (#6).
