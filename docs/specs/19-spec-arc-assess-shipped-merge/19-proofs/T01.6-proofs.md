### Idempotency by construction
Two consecutive /arc-assess runs against the same working tree produce byte-identical Merge Candidates sections because:
1. Inputs (`docs/skill/arc/waves/*.md`, `docs/specs/NN-spec-*/` basenames) are never written by /arc-assess.
2. Merge-candidate classification writes no manifest row, so run 2 sees the same manifest state as run 1.
3. Provenance comments contain no timestamps, no UUIDs, no run counters — only the literal `<!-- aligned-from: ... -->` HTML comment.
4. AskUserQuestion option ordering is deterministic (derives from glob enumeration order, stable per working tree). Given identical user selection, identical row is rendered.
5. Section omission rule (no candidates → no header) is deterministic.

Per SKILL.md line 303 ("Determinism & Idempotency"): "Running it twice in the same working tree produces a byte-identical shipped_spec_index ... The index is not persisted between runs — re-invocations rebuild it from the current state of docs/skill/arc/waves/*.md and docs/specs/NN-spec-*/."
